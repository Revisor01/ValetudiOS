  <files>ValetudoApp/ValetudoApp/Services/SSEConnectionManager.swift</files>
  <action>
NWPathMonitor in den bestehenden `SSEConnectionManager` actor integrieren (per FIX-02). Folgende Aenderungen:

1. **Import hinzufuegen:** `import Network` am Dateianfang (neben `import Foundation` und `import os`).

2. **Neue Properties im Actor:**
```swift
private var pathMonitor: NWPathMonitor?
```

3. **Neue private Methode `startMonitorIfNeeded()`:**
- Erstellt NWPathMonitor nur einmal (guard pathMonitor == nil)
- Setzt `pathUpdateHandler`: Bei `path.status == .satisfied` rufe `reconnectAll()` auf via `Task { await self?.reconnectAll() }`
- Startet Monitor auf eigener DispatchQueue: `DispatchQueue(label: "sse.pathmonitor", qos: .utility)`
- Speichert Monitor in `self.pathMonitor`

WICHTIG zu Pitfall 3: Der Handler feuert sofort beim start() mit dem initialen Pfad. Das ist harmlos, weil `reconnectAll()` nur vorhandene Tasks cancelled — beim ersten Aufruf gibt es noch keine.

4. **Neue private Methode `reconnectAll()`:**
```swift
private func reconnectAll() {
    for key in tasks.keys {
        tasks[key]?.cancel()
        tasks[key] = nil
        isConnected[key] = false
    }
    logger.info("SSE: Network path changed — all connections reset")
}
```
Tasks canceln genuegt. `RobotManager.startRefreshing()` pollt alle 5 Sekunden ob SSE aktiv ist und ruft `connect()` erneut auf. Kein direkter reconnect aus `reconnectAll()` noetig.

5. **`connect()` erweitern:** Am Anfang der bestehenden `connect()`-Methode (vor dem Task-Cancel/Create-Block) `startMonitorIfNeeded()` aufrufen.

6. **`disconnectAll()` erweitern:** Am Anfang `pathMonitor?.cancel()` und `pathMonitor = nil` hinzufuegen, BEVOR die Tasks gecancelled werden.

NICHT aendern: Die `streamWithReconnect()`-Logik, das Backoff-Verhalten, die `disconnect(robotId:)`-Methode. Diese funktionieren korrekt.

Da SSEConnectionManager ein `actor` ist, muss `[weak self]` im `pathUpdateHandler` Closure verwendet werden und der Aufruf ueber `Task { await self?.reconnectAll() }` erfolgen (Handler laeuft auf DispatchQueue, nicht im Actor-Kontext).
