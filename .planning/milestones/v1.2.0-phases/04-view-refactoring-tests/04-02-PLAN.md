Refactor RobotDetailView to be a declarative shell:

1. Replace all extracted @State properties with a single @StateObject:
```swift
@StateObject private var viewModel: RobotDetailViewModel
```

2. Add init that creates the ViewModel:
```swift
init(robot: RobotConfig, robotManager: RobotManager) {
    _viewModel = StateObject(wrappedValue: RobotDetailViewModel(robot: robot, robotManager: robotManager))
}
```
Note: Because RobotDetailView receives robotManager via parameter (not EnvironmentObject for VM init), the calling site in RobotListView/ContentView must pass it. If the View currently gets robotManager from @EnvironmentObject, keep that and pass it to the StateObject init in .onAppear or use a factory pattern. Preferred: pass robotManager in View init.

3. Replace all `$propertyName` bindings with `$viewModel.propertyName`
4. Replace all inline Task { } action closures with viewModel method calls:
   - Button("Start") { Task { try await api.basicControl(...) } } -> Button("Start") { Task { await viewModel.startCleaning() } }
5. Replace .task { ... large loading block } with .task { await viewModel.loadData() }
6. Replace .refreshable { } with .refreshable { await viewModel.refreshData() }
7. Keep ONLY UI-presentation @State that is purely view-local:
   - @State private var showFullMap: Bool (sheet toggle)
   - @State private var showTimers: Bool (sheet toggle)
   - @State private var showUpdateWarning: Bool (alert toggle — unless ViewModel manages it)

8. The View should reference `viewModel.segments`, `viewModel.consumables`, `viewModel.isCleaning`, etc.
9. Keep @EnvironmentObject var errorRouter: ErrorRouter in the View (pass to VM methods as needed, or let VM hold a reference).

The View body structure (sections, layout) stays the same. Only the data source and action handlers change.

Build and verify the app compiles. Run the app in simulator briefly to ensure the detail view still renders (no crash).
