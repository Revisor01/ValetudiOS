## Detailed Findings

### DEBT-01 — FAIL: calculateMapParams exists twice

**Requirement:** The function `calculateMapParams` must exist in exactly one place.

**Finding:** Two independent implementations exist:

1. `Utilities/MapGeometry.swift:21` — canonical free function with `padding` parameter
2. `Views/MapDrawingOverlay.swift:256` — inline method on `MapContentView` extension, identical algorithm but hardcoded `padding = 20`, no `padding` parameter

The comment on line 255 of MapDrawingOverlay.swift says "Delegates to free function in MapGeometry.swift" but the implementation does **not** call the global function — it reimplements the algorithm verbatim. This is a direct violation of DEBT-01.

**Impact:** Risk of divergence if the algorithm changes in MapGeometry.swift but not in MapDrawingOverlay.swift.

---

### DEBT-03 — PASS (with note)

All `try?` occurrences are in:
- Background polling loops (`Task.sleep`, SSE fallback HTTP GET)
- SSE error-recovery paths (not user-initiated)
- JSON encode/decode of persisted data (UserDefaults, Keychain helpers)
- `Task.sleep` calls (cancellation-safe pattern)

The one case in `RobotDetailSections.swift:261` (`robotProperties = try? await p`) is in a parallel async-let load during view `onAppear` — a background fetch, not a user tap. Similarly `MapView.swift:148` (`restrictions = try? await restrictionsTask`) is in initial load.

No user-tap action uses `try?` to silently discard errors.

### TEST-03 — FAIL: SSE backoff unit tests missing

**Requirement:** Unit tests for SSE backoff logic.

**Finding:** No test file with SSE, backoff, or retry logic found. The backoff is implemented in `MapViewModel.swift` (exponential retry loop in `startSSEStreaming()`), but no corresponding test exists.

### TEST-04 — FAIL: MapCacheService unit tests missing

**Requirement:** Unit tests for MapCacheService.

**Finding:** `MapCacheService.swift` implements `saveIfChanged()` with hash-based write prevention, but no test file (e.g., `MapCacheServiceTests.swift`) exists.

---

### ROBUST-02 — FAIL: Confirmation dialogs incomplete

**Requirement:** All destructive actions have confirmation dialogs.

**Finding:**

- `ConsumablesView.swift:46` — `confirmationDialog` exists for consumable reset. PASS.
- `RobotSettingsView.swift:445–461` — Map reset and mapping pass use `.alert` (not `confirmationDialog`). This technically shows a confirmation, but `.alert` on iOS is less discoverable than `.confirmationDialog`. Whether this satisfies the requirement depends on interpretation.
- `SettingsView.swift:266` — Delete robot uses `Button(role: .destructive)` inside a List swipe action with no separate confirmation dialog.

The delete-robot action in SettingsView has no confirmation step — a destructive action with no undo.

## Gap Summary

### Gap 1: DEBT-01 — Duplicate calculateMapParams

`MapDrawingOverlay.swift` has its own inline reimplementation instead of calling `calculateMapParams(layers:pixelSize:size:padding:)` from `MapGeometry.swift`.

**Fix:** Replace the method body with a call to the global function:
```swift
func calculateMapParams(layers: [MapLayer], pixelSize: Int, size: CGSize) -> MapParams? {
    ValetudoApp.calculateMapParams(layers: layers, pixelSize: pixelSize, size: size, padding: 20)
}
```
Or remove the wrapper entirely and call the global function directly at the two call sites.

### Gap 2: TEST-03 — No SSE backoff tests

Create `ValetudoAppTests/SSEBackoffTests.swift` testing the retry delay progression in `MapViewModel.startSSEStreaming()`.

### Gap 3: TEST-04 — No MapCacheService tests

Create `ValetudoAppTests/MapCacheServiceTests.swift` testing:
- `saveIfChanged` skips write when data hash is unchanged
- `saveIfChanged` writes when data changes
- `load` returns nil when no cache exists

### Gap 4: ROBUST-02 — Delete robot has no confirmation

`SettingsView.swift:266` performs robot deletion via swipe action with `Button(role: .destructive)` but no confirmation step. Add a `confirmationDialog` or `.alert` before deletion.
